# 4강 : 오류 처리 - 작성자: 한지윤

# 1️⃣ 소개

C++에서는 단순히 `if` , `while`  같은 문장으로 프로그램을 만드는 것이 아니라 
**타입(type), 알고리즘(algorithm), 추상화(abstraction)**를 활용해 오류를 줄이고 더 안전한 프로그램을 만듦
→ 이미 검증된 라이브러리를 사용하여 실수 가능성을 줄인다

<aside>

#### 타입, 알고리즘, 추상화

- **타입(Type)**
    - `int` , `double`, `string` 같은 데이터 종류 → **데이터의 형식**
- **알고리즘(Algorithm)**
    - 데이터를 어떻게 처리할지 정해진 방법
    - `sort()` , `find_if()`
- **추상화(Abstraction)**
    - 내부 구조를 숨기고 쉽게 사용할 수 있게 만든 것
    → 내부의 복잡함을 숨기고 기능만 사용
    - `vector` , `string`, `map`, `thread`

#### 예제: 전통방식 vs 최신방식

**전통적인 방식의 코드**

```cpp
int arr[5] = {3, 1, 4, 2, 5};

// 직접 정렬 (실수 가능)
for (int i = 0; i < 5; i++) {
    for (int j = i + 1; j < 5; j++) {
        if (arr[i] > arr[j]) {
            int tmp = arr[i];
            arr[i] = arr[j];
            arr[j] = tmp;
        }
    }
}
```

**C++방식 (추상화+알고리즘)**

```cpp
#include <vector>    // 안전한 배열
#include <algorithm> // 검증된 알고리즘
#include <iostream>

int main() {
    std::vector<int> v = {3, 1, 4, 2, 5};

    std::sort(v.begin(), v.end());  // 검증된 알고리즘 사용

    for (int x : v)
        std::cout << x << " ";
}
```

</aside>

이 같은 고급 구조체는 프로그래밍을 간소화하고, 실수의 가능성을 줄이고, 컴파일러가 오류를 잡을 가능성을 높임

프로그램이 커질수록 오류 처리 표준이 중요해진다

# 2️⃣ 예외

예외(Exception)는 프로그램 실행 중 발생하는 오류를 안전하게 처리하기 위한 C++의 메커니즘

### 예제코드

```cpp
std::vector<int> v = {1,2,3}; // 크기가 3인 배열
v[5];  // 존재하지 않는 인덱스 접근 -> 범위 밖 접근
```

### 🤔 뭐가 문제일까….

- Vector를 만든 사람은 **이걸 사용하는 사람이 어떤 상황에서 쓸지 모름**
- 사용하는 사람은 **오류를 매번 직접 체크하기 어려움**

### 🧐 해결 방법 = 예외(Exception)

- 문제가 발생하면, **알려주고 책임을 넘김**

**동작 흐름**

1. **문제가 발생한 것을 던짐**
    
    ```cpp
    throw out_of_range("범위 초과!"); // 던졌다!
    ```
    
2. **호출한 쪽으로 전달됨 → 함수 호출 스택을 따라 올라감**
    
    ```cpp
    main()
     └── f()
          └── g() 
               └── h() ← 여기서 throw
    ```
    
    `catch`문을 만날때까지 올라감 
    → 처리 가능한 위치를 찾을 때 까지 계속 올라감
    
3. **try-catch에서 잡음**
    
    ```cpp
    try {
        // 문제 발생 가능 코드
    }
    catch (...) {
        // 처리
    }
    ```
    
    **왜 이렇게 할까..**
    
    - 기존 방식대로 if문으로 하게되면 모든 곳에서 체크 필요함 → 불필요한 연산이 늘어남
    - `throw + try/catch` 로 하게 될 경우 → 한 곳에서 처리 가능
        - 책임의 분리가 가능하다~
            - 문제 발생: 라이브러리
            - 문제 처리: 사용자코드
    - 프로그램 안정성 증가: 예상 못한 오류에도 안전하게 대응 가능
    - 대형 프로젝트에서 필수적: 함수가 수십 개 연결될 대 → 아래에서 발생한 오류를 위에서 처리

### 예제 코드

```cpp
double& Vector::operator[](int i)
{
    if (!(0 <= i && i < size()))
        throw out_of_range{"Vector::operator[]"}; **// 범위 벗어남**

    return elem[i];
}
```

- `throw`는`Vector::operator[]` 를 직접적으로 혹은 간접적으로 호출했던 함수에서 `out_of_range` 타입의 예외를 처리하는 핸들러를 찾아 제어를 넘긴다

**예외처리부분**

```cpp
void f(Vector& v) {
	try { // 에러가 일어날 것 같은 함수를 쓴다
    compute1(v); 
    compute2(v);
	}
	catch (const out_of_range& err) { **// 범위 벗어났을 때 처리**
	    cout << err.what(); **// 예외 메시지 출력**
	}
}
```

- 피호출자에서 예외가 발생한 경우, 함수 호출 스택을 풀고 호출자의 컨택스트로 돌아간다
    - 컨택스트: 현재 작업중인 위치, 피호출자가 호출된 위치를 의미함
- 이 과정에서 피호출자에서 소멸자가 호출될 수 있다
    - 예외가 발생할 경우 함수 호출 스택을 풀게되는데, 이 과정에서 피호출자에서 생성된 지역 변수들이 소멸되는 것을 의미한다

## ⚠️ 주의할 점

1. try 남용 금지
    - 성능 저하
    - 코드가 복잡해짐
2. 모든 함수에서 예외를 처리하지 말것
    - 가장 좋은 방법은 최상위(메인 or 컨트롤 레이어)에서 처리
3. 참조로 catch 해야함
    - 복사하면 성능 손실 발생
    
    ```cpp
    catch (const out_of_range& e)
    ```
    

<aside>

**예외 처리 흐름**

- 문제 발생 → throw → 호출 스택 이동 → catch에서 처리

**좋은 설계**

- 오류 발생: 하위 레이어
- 오류 처리: 상위 레이어
</aside>

## 📌 RAII(Resource Acquisition Is Initialization)

해당 항목을 보면 RAII 패턴을 사용하라고되어 있음

### 개념

- 자원은 객체 생성 시 얻고, 객체가 사라질 때 자동으로 정리하는 C++ 설계 방식

### 문제상황

```cpp
void f() {
    int* p = new int[10];
    throw ...;  // 여기서 터짐
    delete[] p; // 실행 안됨 → 메모리 누수
}
```

### RAII 방식

```cpp
#include <vector>

void f() {
    std::vector<int> v(10);
    throw ...;
}
```

- 예외 발생해도 vector 소멸자 호출됨 → 자동으로 메모리 정리

이것과 연결되는 개념으로 `new` , `delete` 를 사용하기 보다는 
스마트 포인터(unique_ptr, shared_ptr)나 `vector`, `string` 같은 STL을 이용하도록 하자

→ 소멸될때 소멸자가 알아서 호출되도록 만들자

# 3️⃣ 불변

**불변**

- 항상 지켜져야 하는 상태
- 객체가 항상 유지해야하는 상태
- 예시:
    
    `elem` → 실제 배열
    `sz` → 크기
    
    ∴ elem은 sz개의 원소를 가르킨다 → 불변
    
    - elem은 5개인데 sz는 10개 → 말도 안될뿐더러 위험함

**전제조건**

- 함수가 실행되기 전 반드시 만족해야 하는 조건
- 함수가 정상적으로 동작하려면 반드시 만족해야하는 조건
- 예시: `v[i]` 에서 `0 <= i < size()` 부분은 반드시 만족해야하는 **전제조건**

## 🧐 왜 중요할까?

1. 오류를 초기에 잡을 수 있음
    
    ```cpp
    Vector v(-27);  // 잘못된 입력 -> 바로 예외 발생
    ```
    
2. 이상 상태 방지
    - 불변이 깨지면 데이터가 깨지고 프로그램이 오작동하게 됨
3. 코드 설계 기준 제공
    - 이 함수는 무엇을 보장해야 하는가?
4. 예외 처리와 연결된다
    - 조건이 안 맞으면: `throw → 위로 전달 → 처리`

### 예제

```cpp
class Vector {
    double* elem;
    int sz;

public:
    Vector(int s) {
        if (s < 0)
            throw std::length_error("negative size"); //v[-27]

        elem = new double[s]; // v[1'000'000'000]
        sz = s;
    }
};
```

위 코드에서 발생할 수 있는 에러는 아래와 같이 처리 할 수 있다

```cpp
void test(int n) {
    try {
        Vector v(n);
    }
    catch (std::length_error&) {
        cout << "크기가 잘못됨" << endl;
    }
    catch (std::bad_alloc&) {
        cout << "메모리 부족" << endl;
    }
}
```

<aside>

잘 디자인된 코드에는 `try` 블록이 거의 없다
RAII 기법을 체계적으로 사용함으로써 남용을 막자!

</aside>

# 4️⃣ 오류 처리 대안

오류는 상황에 따라 `예외`, `에러 코드`, `프로그램 종료`중 적절한 방법으로 처리해야 함

## **🔎 오류 처리 방법 3가지**

- 예외 던지기(throw)
- 실패를 나타내는 값 반환(error code)
- 프로그램 종료(terminate(), exit(), abort() 등)

### 각각 언제 쓰는가?

1. 에러 코드를 반환하는 경우
    
    ```cpp
    bool openFile() {
        return false;  // 실패
    }
    ```
    
    - 실패가 정상적으로 발생할 수 있는 경우
    - 예시
        - 파일 열기 실패
        - 네트워크 끊김
        - 사용자 입력 오류
2. 예외를 사용하는 경우
    
    ```cpp
    throw std::out_of_range("index error");
    ```
    
    - 오류가 비정상적이고 드물 때
    - 호출한 함수가 처리하지 못할 때
    - 예시
        - 범위 밖 접근
        - 논리적으로 불가능한 상황
3. 프로그램 종료
    
    ```cpp
    std::terminate();
    ```
    
    - 복구가 불가능한 상황
    - 예시
        - 메모리 완전 고갈
        - 시스템에 치명적인 오류
    - 확실히 종료시키는 한 가지 방법은 함수에 `noexcept` 를 추가해 함수 구현 내 어디에서든 `throw`가  `terminate()`로 바뀌도록 하는 것
        - `noexcept` : 이 함수는 절대로 예외(throw)를 발생시키지 않는다고 컴파일러와 약속하는 키워드
        - 이 함수가 예외를 절대 안던진다고 약속했는데 예외가 발생한다? 
        걍 프로그램 종료가 답인거임

# 5️⃣ 어설션

“이 조건은 반드시 참이어야한다”라고 프로그램에 명시적으로 검사하는 도구

- 대규모 프로그램에서 테스트 중 런타임 검사 시(디버깅 시) 사용자에게 여기서는 이 값이 반드시 이것이 되어야 한다고 명시해주는 도구

## 1. assert()

- 표준 라이브러리에서 제공하는 디버그 매크로
- 런타임에 참이어야 하는 조건을 표명(assert)함

```cpp
void f(const int n) {
	assert(n >= 0); // n은 음수여서는 안된다
}
```

- `디버그 모드` 에서 `assert()` 의 조건이 실패하면 프로그램은 종료됨
    - 디버그 모드가 아닌 경우 assert는 검사되지 않음
    - 다소 조잡하고 유연하지 못하지만 아무것도 안하는 것 보다 나음

## 2. 정적 어설션(static_assert)

- 런타임에서 오류를 찾아서 예외를 보고하는 것보다 컴파일 타임에서 미리 찾는 편이 좋음
- 컴파일할 때 이 조건이 맞아야 한다고 보고하는 것
- 틀리면 컴파일 자체가 되지 않음

```cpp
constexpr double C = 299792.458; // km/s

void f(double speed) {
	constexpr double local_max = 160.0/(60*60) // 160km/h
	
	static_assert(speed < C, "해당 속도로 갈 수 없습니다");    // 안됨
	static_assert(local_max < C, "해당 속도로 갈 수 없습니다") // 가능
} 
```

- 예제에서 `speed` 는 런타임 시점에 들어오는 값 이므로 컴파일 때 알 수 없으므로 `static_assert`사용 불가능
- `local_max` 는 `constexpr` 값 이므로 계산 결과가 컴파일 시점에 확정되어서 `static_assert`사용 가능
    - `const`는 값이 바뀌지 않지만 컴파일 타임에 값이 보장되지 않으므로 `static_assert` 사용 불가능

## 3. noexcept

이 함수는 예외를 절대 던지지 않는다.

- 빠르게 동작 가능하며 예외 처리를 준비하지 않아도 되므로 성능 최적화에 적합
- 절대 실패하지 않는다는 확신이 가능하므로 안정성이 보장됨
- 하지만 남용하면 당연히 안좋음
